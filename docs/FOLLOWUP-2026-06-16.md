## 1. Checkpoint restore: notes re-anchor by content (was deferred call #2)

**The problem.** A checkpoint restore (`Document::restore_state`) replaces the
whole text as one wholesale op. Margin notes are an overlay anchored by char
range, not part of the text stream, and checkpoints store only `(text, spans,
blocks)` — never the annotation overlay. The wholesale delete-all+insert-all
ran through the ordinary note-adjustment math (`Annotations::apply_op`), whose
documented "a fully deleted anchor collapses to a point" rule fired for *every*
note at once: they all piled up as zero-width points at the very end of the
restored text. Bodies/statuses/count survived and undo brought them back, but
the on-screen result — every card stacked at the bottom — was surprising.

**The decision (Kirill, Option A): keep notes, re-anchor by content.** A note
should *follow the passage it covers* into the restored text, exactly the way a
diagnosis anchors to its quote. If the passage is gone, the note detaches
honestly — flagged, parked near where it was — rather than silently sitting at
a wrong place or piling at the end.

**Implementation** (`document.rs`):

- New `Annotation::orphaned: bool` (`#[serde(default)]`, so old `.strop` files
  load unchanged). Set *only* by re-anchoring; ordinary editing never touches
  it. It persists, so a saved-after-restore document reopens in the same state,
  and it round-trips through undo (the pre-restore snapshot has it `false`).
- New `Annotations::reanchor(old_text, new_text)`: for each note, take the
  substring it covered in the *old* text and locate it in the *new* text via
  `diagnose::anchor`, searching from the note's own former offset so repeated
  passages resolve in document order (the same trick `to_annotations` uses for
  repeated quotes). Found → follow; gone (or a zero-width point with nothing to
  match) → `orphaned = true`, range clamped to a point at the former offset.
- `restore_state` captures `old_text` and computes the re-anchoring *before*
  the wholesale swap erases it, then installs the result after the swap (the
  swap's apply_op mangling of notes is discarded, like it already is for
  spans/blocks).

Because the app reads `doc.notes()` at both render and save time, no app wiring
was needed for the data to flow — but a card whose anchor is a guess must not
*read* as confidently anchored, so the margin/drawer card labels gain a quiet
"· detached" suffix when `orphaned` (no geometry/colour/layout change — the
hand-tuned card rendering is otherwise untouched). `MarginCard` carries the
flag; `note_card_label()` builds the label for both the wide margin and the
narrow drawer.

**Behaviour preserved.** `restore_state_is_one_undoable_transaction` (no notes)
is unchanged; the save→reopen round-trip property still holds (orphaned stays
false on both sides). Four new unit tests pin the new behaviour: follows a
moved passage, keeps repeated passages in order, detaches a vanished passage
*without* piling at the end, and undoes in one step clearing the flag.

**Rejected alternatives.** (B) snapshotting the annotation overlay per
checkpoint — faithful, but a restore would then *delete* notes made later
(surprising in the other direction) and bloats every checkpoint. (C) clamping
old offsets blindly — cheap, but points notes at wrong text when versions
diverge, with no signal that anything moved. (Diff-based restore — apply a
minimal diff so notes ride the existing per-edit anchor math — is elegant but a
much larger change to a load-bearing path; content re-anchoring gets the same
"notes follow their text" result far more cheaply.)

## 2. Single-instance / already-open detection

**The problem.** Strop is "one document, one window, one process" and the
durable `.strop` is a Loro store with no multi-process merge. Opening a file a
second time spawned a *second writer* on the same store — the multiplayer-
persistence hole Kirill wanted closed. Ideal outcome: the existing window
surfaces instead.

**Design** (`single_instance.rs`). The first instance to open a file binds a
per-file Unix socket under `$XDG_RUNTIME_DIR` (keyed by the file's *canonical*
path, hashed to stay under the `sun_path` limit) and becomes its PRIMARY. A
later instance:

1. **connects** to that socket. If something live answers, it sends a one-byte
   "raise" and reports `AlreadyOpen` — the caller exits *before* opening (and
   mutating) the store, so there is never a second writer.
2. if nothing answers, it **unlinks any stale socket and binds** — becoming the
   new primary.

**The safety property: liveness is the socket, never a lock file.** A crashed
primary leaves a stale socket the next launch simply rebinds, so a dead
instance can *never* lock you out of your own document — the one failure mode a
naive `flock` would introduce, and a far worse regression than the bug being
fixed. The only cost is a microscopic simultaneous-launch race (both fail to
connect, both bind); the bind loser reconnects and hands off, and a truly
concurrent bind just opens two windows — exactly today's behaviour, so never a
regression.

**Wiring** (`main.rs`). The claim runs right after the store path is resolved
and *before* `Store::open`, gated on `!smoke` (smoke runs never claim). On
`AlreadyOpen`: print a line and `std::process::exit(0)` — nothing opened,
nothing to flush. On `Primary`: hold the guard for the process lifetime in a
foreground task that polls it on a 400 ms timer and calls `cx.activate(true)`
when a hand-off arrives (the proven background→foreground bridge from
`smoke.rs`). The guard's `Drop` stops the accept loop and unlinks the socket.

**Raising the window is best-effort.** `cx.activate(true)` is what the app
already uses to surface itself at launch; on Wayland the backend maps it to
xdg-activation where the compositor honours it. If the compositor ignores it,
the worst case is the second instance exits quietly without a visible raise —
still strictly better than two writers corrupting one store.

**Tested.** Four unit tests in `single_instance.rs` cover the protocol end to
end without a GUI: socket-path stability/uniqueness, primary→secondary
detection + raise signalling, **stale-socket takeover (never locks the file)**,
and socket release on guard drop. The `main.rs` glue is small and clippy-clean.

**Manual GUI check (for the dev rig — the live raise can't be asserted
headlessly):** open a document, then run `strop <that-same-file>` again; the
second process should print `… is already open — raising that window` and exit
immediately (no second window), and the first window should come forward where
the compositor honours activation.

### Limits / not done

- Same *file*, not same *content*: two different paths pointing at the same
  inode via symlink resolve to the same canonical path, so they share a socket
  (good); hardlinks do not (each link is its own canonical path → two writers
  still possible, as before). Acceptable — hardlinked `.strop` files are not a
  supported workflow.
- The hand-off only says "surface"; it does not pass a *new* file to an
  existing window (Strop is one-doc-per-process, so there is no "open this too"
  to forward). Opening a *different* file always starts its own process, as
  before.
