## 1. Checkpoint restore: notes re-anchor by content (was deferred call #2)

**The problem.** A checkpoint restore (`Document::restore_state`) replaces the
whole text as one wholesale op. Margin notes are an overlay anchored by char
range, not part of the text stream, and checkpoints store only `(text, spans,
blocks)` — never the annotation overlay. The wholesale delete-all+insert-all
ran through the ordinary note-adjustment math (`Annotations::apply_op`), whose
documented "a fully deleted anchor collapses to a point" rule fired for *every*
note at once: they all piled up as zero-width points at the very end of the
restored text. Bodies/statuses/count survived and undo brought them back, but
the on-screen result — every card stacked at the bottom — was surprising.

**The decision (Kirill, Option A): keep notes, re-anchor by content.** A note
should *follow the passage it covers* into the restored text, exactly the way a
diagnosis anchors to its quote. If the passage is gone, the note detaches
honestly — flagged, parked near where it was — rather than silently sitting at
a wrong place or piling at the end.

**Implementation** (`document.rs`):

- New `Annotation::orphaned: bool` (`#[serde(default)]`, so old `.strop` files
  load unchanged). Set *only* by re-anchoring; ordinary editing never touches
  it. It persists, so a saved-after-restore document reopens in the same state,
  and it round-trips through undo (the pre-restore snapshot has it `false`).
- New `Annotations::reanchor(old_text, new_text)`: for each note, take the
  substring it covered in the *old* text and locate it in the *new* text via
  `diagnose::anchor`, searching from the note's own former offset so repeated
  passages resolve in document order (the same trick `to_annotations` uses for
  repeated quotes). Found → follow; gone (or a zero-width point with nothing to
  match) → `orphaned = true`, range clamped to a point at the former offset.
- `restore_state` captures `old_text` and computes the re-anchoring *before*
  the wholesale swap erases it, then installs the result after the swap (the
  swap's apply_op mangling of notes is discarded, like it already is for
  spans/blocks).

Because the app reads `doc.notes()` at both render and save time, no app wiring
was needed for the data to flow — but a card whose anchor is a guess must not
*read* as confidently anchored, so the margin/drawer card labels gain a quiet
"· detached" suffix when `orphaned` (no geometry/colour/layout change — the
hand-tuned card rendering is otherwise untouched). `MarginCard` carries the
flag; `note_card_label()` builds the label for both the wide margin and the
narrow drawer.

**Behaviour preserved.** `restore_state_is_one_undoable_transaction` (no notes)
is unchanged; the save→reopen round-trip property still holds (orphaned stays
false on both sides). Four new unit tests pin the new behaviour: follows a
moved passage, keeps repeated passages in order, detaches a vanished passage
*without* piling at the end, and undoes in one step clearing the flag.

**Rejected alternatives.** (B) snapshotting the annotation overlay per
checkpoint — faithful, but a restore would then *delete* notes made later
(surprising in the other direction) and bloats every checkpoint. (C) clamping
old offsets blindly — cheap, but points notes at wrong text when versions
diverge, with no signal that anything moved. (Diff-based restore — apply a
minimal diff so notes ride the existing per-edit anchor math — is elegant but a
much larger change to a load-bearing path; content re-anchoring gets the same
"notes follow their text" result far more cheaply.)
